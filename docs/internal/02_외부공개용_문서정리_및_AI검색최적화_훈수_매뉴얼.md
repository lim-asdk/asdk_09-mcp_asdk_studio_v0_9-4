# 02. 외부공개용 문서정리 및 AI검색최적화 훈수 매뉴얼

작성 목적:

- 이 문서는 `mcp_asdk_studio_v1` 프로젝트의 코드 구현이 아니라 문서 운영 방식에 대한 내부 훈수용 정리다.
- 핵심은 "외부 공개 문서를 어떻게 정리해야 AI와 사람 둘 다 잘 찾고, 잘 이해하고, 잘 인용하는가"에 맞춘다.
- 특히 지금은 일반 검색보다도 AI가 문서를 많이 읽고 요약하고 응용하는 시대라는 전제를 강하게 둔다.

## 1. 문서 운영의 기본 원칙

이 프로젝트 문서는 처음부터 세 층으로 나눠서 운영하는 것이 좋다.

### A. 외부 공개용 문서

목적:

- GitHub 방문자
- 검색엔진
- AI 요약 시스템
- 도입 검토자
- 실제 사용자

이 문서는 공개 저장소 기준으로 안전해야 한다.

### B. 내부 개발용 문서

목적:

- 설계 메모
- 우선순위
- 구현 결정 이유
- 기술 부채
- 다음 할 일

이 문서는 협업과 인수인계용이다.

### C. 로컬 전용 민감 문서

목적:

- 실제 키 값
- 실제 고객/개인 환경 정보
- 배포 비밀값
- 외부 공개 전 논의 메모

이 문서는 Git에 올리지 않는 것이 원칙이다.

## 2. 가장 중요한 판단 기준

외부 문서는 "예쁘게 쓰는 문서"보다 "AI와 사람이 빠르게 정확히 이해할 수 있는 문서"가 더 중요하다.

즉, 공개 문서는 다음 조건을 만족해야 한다.

- 프로젝트 이름이 분명하다
- 이 프로젝트가 무엇인지 첫 문단에서 바로 알 수 있다
- 어떤 사용자를 위한 것인지 명확하다
- 어떤 기능이 있는지 구조적으로 보인다
- 어떻게 실행하는지 빠르게 찾을 수 있다
- 어떤 기술 스택인지 분명하다
- 어떤 부분이 공개 안전하고 어떤 부분이 로컬 전용인지 명확하다

## 4. 왜 영어와 국문이 둘 다 중요한가

### 한국어 문서의 역할

- 실제 한국 사용자 유입
- 국내 협업자와 운영자 인수인계
- 국내 커뮤니티 확산

### 영어 문서의 역할

- GitHub 전 세계 검색 대응
- 해외 사용자 유입
- AI 시스템의 기본 인용 소스 확보
- 기술 블로그, 튜토리얼, 링크 공유 시 기본 기준점 확보

즉, 영어 문서는 "해외용"만이 아니라 "AI에게 읽히는 기본 문서"라는 관점으로 보는 것이 좋다.

## 5. AI가 잘 찾고 잘 인용하게 만드는 문서 작성법

AI가 문서를 잘 찾고 응용하게 하려면 문서가 아래 특성을 가져야 한다.

### 제목은 모호하면 안 된다

좋은 예:

- `What is MCP ASDK Studio`
- `How to configure MCP servers`
- `How to set up AI provider keys safely`
- `Architecture overview for desktop and browser runtime`

나쁜 예:

- `Notes`
- `Guide`
- `Memo`
- `Reference`

### 첫 문단에서 정체를 바로 말해야 한다

문서 첫 3~5줄 안에 아래가 들어가면 좋다.

- 프로젝트가 무엇인지
- 어떤 문제를 해결하는지
- 어떤 사용자 대상인지
- 데스크톱/브라우저/서버 중 무엇을 지원하는지

### 리스트와 구조를 분명히 해야 한다

AI는 구조가 분명한 문서를 더 잘 요약한다.

좋은 방식:

- 섹션 제목
- 짧은 설명
- 불릿 리스트
- 표준 용어 반복

### 용어를 일관되게 써야 한다

예를 들어 아래 용어는 문서마다 섞지 말고 기준을 정하는 것이 좋다.

- `AI Profile`
- `MCP Server`
- `Workspace`
- `Desktop Runtime`
- `Browser Runtime`
- `Local-only Config`
- `Public Template`

## 6. 공개용 README에서 반드시 보여야 하는 것

`README.md`는 가장 중요하다.
AI와 사람 모두 거의 여기서 시작한다.

README에는 최소한 아래가 있어야 한다.

1. 제품 한 줄 설명
2. 주요 기능 요약
3. 지원 실행 모드
4. 빠른 시작
5. 설정 파일 위치
6. 비밀값 처리 원칙
7. 스크린샷 또는 UI 이미지
8. 문서 링크
9. 라이선스와 보안 정책 링크

README 첫 부분에 들어가면 좋은 문장 예시는 아래와 같다.

`MCP ASDK Studio is a desktop-first AI workspace that lets users configure their own AI providers, connect MCP servers, inspect tools, and run chat-driven workflows through a product-style interface.`

이런 문장은 AI가 프로젝트 성격을 요약할 때 매우 잘 활용한다.

## 7. 공개 문서에서 주석과 설명을 쓰는 방식

외부 공개 문서는 코드 주석처럼 쓰면 안 된다.
반대로 너무 마케팅 문구만 있어도 안 된다.

균형은 다음과 같다.

- 제품 소개 문서는 사람과 AI가 빠르게 이해할 수 있게 명확하게
- 설정 문서는 실행 가능한 정보 위주로
- 아키텍처 문서는 과장 없이 실제 구조 중심으로
- FAQ는 사용자가 실제로 부딪히는 질문 중심으로

좋은 문서의 특징:

- 문장 길이가 너무 길지 않다
- 한 문단에 주제가 하나다
- 추상어보다 구체어를 쓴다
- 파일명, 명령어, 경로가 분명하다

## 8. 내부 문서 파일명 규칙 추천

내부 문서는 한글 파일명으로 운영하는 것이 좋다.

추천 예시:

## 10. 공개하면 안 되는 정보

아래 정보는 절대 공개 문서에 넣지 않는다.

- 실제 API 키
- 실제 토큰
- 개인 PC 경로
- 실제 계정명
- 개인 이메일
- 비공개 서버 주소
- 내부 판단 메모
- 고객 정보

주의할 점은 "문서"만 조심하면 되는 것이 아니라는 것이다.
아래도 함께 조심해야 한다.

- 예제 JSON
- 스크린샷
- 로그 캡처
- 코드 주석
- 커밋 메시지
- PR 설명

## 11. 커밋과 푸시 전에 반드시 볼 체크리스트

공개 저장소에 푸시하기 전에 아래를 반드시 확인한다.

1. `.env`가 Git에 안 들어가는지 확인
2. `user_data`, `logs`, `tmp`, `screenshots_private` 같은 로컬 폴더가 제외되는지 확인
3. 공개 예제 JSON에 실제 키가 없는지 확인
4. 스크린샷에 개인 정보, 경로, 토큰이 없는지 확인
5. 커밋 메시지에 민감한 정보가 없는지 확인
6. `git diff --cached`로 마지막 staged 내용 확인
7. `git status`로 의도한 파일만 올라가는지 확인

## 12. 공개용 예제 파일을 만들 때 원칙

예제 파일은 "실행 감"이 있어야 하지만 "실제 비밀값"은 없어야 한다.

좋은 방식:

- `.env.example`
- `profile.template.json`
- `servers.example.json`

나쁜 방식:

- 실제 키가 들어간 `config.json`
- 개인 경로가 박힌 실행 예제
- 특정 사용자 계정명을 그대로 보여주는 샘플

## 13. AI 인용과 검색을 위해 더 챙기면 좋은 것

아래는 있으면 확실히 좋다.

- 프로젝트 한 줄 설명이 문서마다 반복됨
- 동일한 핵심 용어를 여러 문서에서 일관되게 사용
- 기능 목록이 README와 PRODUCT_OVERVIEW에 모두 존재
- 실행 흐름이 QUICK_START와 FAQ에 모두 존재
- 데스크톱/브라우저/서버 모드가 문서마다 같은 이름으로 설명됨

AI는 이런 반복과 일관성을 통해 프로젝트 개념을 더 안정적으로 잡는다.

## 15. 내부용과 외부용을 분리하는 운영 팁

가장 안전한 운영 방식은 아래 셋을 분리하는 것이다.

- 공개 저장소
- 비공개 내부 저장소 또는 내부 문서 폴더
- 로컬 전용 민감 데이터 폴더

그리고 역할도 분리한다.

- 외부 공개 문서: 설명, 설치, 사용법, 구조
- 내부 문서: 판단 이유, 우선순위, 미완료 이슈
- 로컬 민감 자료: 키, 실사용 경로, 실제 환경

## 16. 이 문서를 실제로 어떻게 써먹을 것인가

이 문서는 당장 코드 수정용이 아니라 문서 정리 기준표로 쓰는 것이 좋다.
즉, `mcp_asdk_studio_v1` 프로젝트에서 문서 정리를 할 때 아래 순서로 적용한다.

1. 공개해도 되는 문서와 안 되는 문서를 먼저 분리
2. 공개 문서는 영어와 국문 세트를 맞춤
3. README와 PRODUCT_OVERVIEW부터 먼저 정리
4. QUICK_START와 FAQ를 붙임
5. 예제 설정 파일과 보안 문서를 마지막에 점검

## 17. 한 줄 결론

이 프로젝트의 공개 문서는 "사람이 읽기 좋은 문서"를 넘어서 "AI가 쉽게 찾고 안정적으로 요약하고 인용할 수 있는 구조화된 문서"를 목표로 만들어야 한다.
